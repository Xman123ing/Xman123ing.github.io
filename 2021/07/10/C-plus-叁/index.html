14、内存对齐

为什么需要内存对齐？
这是因为不是所有硬件平台都能够访问任意位置的内存，考虑到CPU处理内存的方式，比如32位的CPU，一个时钟周期最多可以处理4byte的内存空间，且起始地址一定是偶数。
假如将一个4byte的整型存放在奇数内存起始位置上，想把这4个字节读出来，32位的CPU就需要读取两次。
但对齐之后，只需要一次读取即可，因为内存对齐之后，数据在内存中的大小将会是4的整数倍。同时此种方式也伴随一个问题，那就是数据在内存之中的存放将不是紧挨着的，而是会出现空隙，这对类似结构体而言，使用sizeof不一定会得到预期的结果。

  • 内存对齐规则由操作系统 + 编译器共同决定；
  • 默认对齐系数：32位机一般为4，64位机一般为8；

14.1、#pragma pack(k)和#pragma pack()

k表示对齐系数，k可以是1,2,3,4,8,16

#pragma pack(1) //内存对齐设置为1个字节

　　struct s1
　　{
　　　　int i;
　　　　char c;
　　　　bool f;
　　}

    //struct s2{...}

　  //...

#pragma pack()   //恢复默认的内存对齐（与文件开头的指令配对使用）

14.2、、#pragma pack(push,1)和#pragma pack(pop)

　#pragma pack(push,1) //内存对齐设置为1个字节

　　struct s3
　　{
　　　　int i;
　　　　char c;
　　　　bool f;
　　}

//struct s4{...}

　  //...

　#pragma pack(pop)   //恢复默认的内存对齐(与文件开头的指令配对使用)

14.3、C++11新增对齐关键字

#include ＜iostream＞
using namespace std;
//自定义的ColorVector， 对齐到32字节的边界
struct alignas(32)ColorVector{
double r;
double g;

};
int main(){
//使用C++11中的alignof来查询ColorVector的对齐方式
cout＜＜"alignof(ColorVector):"＜＜alignof(ColorVector)＜＜endl;

}

alignas(32)设置ColorVector对齐值为32字节；
alignof(ColorVector)获取ColorVector的对齐系数；

15、union共用体/联合体

union的外形看起来与struct非常类似，但二者的区别确实非常明显的

  • struct的成员之间是共存的关系，编译器必须要为每一个成员分配内存（占位符除外，比如零长度的数组），不管是否被使用；
  • union的成员是互斥的关系，只有一个成员’有效’；

本质：
所谓union就是在内存中申请一块足够大的内存，即占据内存最大的那个成员类型的空间，然后所有成员类型均使用相同的首地址

union的使用场合：
是各数据类型各变量占用空间差不多并且对各变量同时使用要求不高的场合

借由union的特性可以判断大小端属性，大小端概念详见<计算机系统.md第3节>

#include<iostream>
using namespace std;
union TEST{
    short a;
    char b[sizeof(short)];
};
int main(){
    TEST test;
    test.a=0x0102;// 不能引用共用体变量，只能引用共用体变量中的成员。
    if(test.b[0]==0x01&&test.b[1]==0x02){
        cout<<"big endian."<<endl;
    }
    else if(test.b[0]==0x02&&test.b[1]==0x01){
        cout<<"small endian."<<endl;
    }
    else{
        cout<<"unknown"<<endl;
    }
    return 0;
}

16、enum枚举类型

16.1、普通枚举类型

enum Months{Jan, Feb, Mar, Apr, May};

普通枚举类型的缺陷

16.1.1、缺陷一：enum成员的名字是全局可见的

C/C++的enum有个很”奇怪”的设定， 就是具名（有名字） 的enum类型的名字，以及enum的成员的名字都是全局可见的。这与C++中具名的namespace、class/struct及union必须通过”名字::成员名”的方式访问相比是格格不入的（namespace等被称为强作用域类型，而enum则是非强作用域类型），比如：

enum Type{General,Light,Medium,Heavy};
enum Category{General,Pistol,MachineGun,Cannon};

Category中的General和Type中的General都是全局的名字，编译时会报错。

16.1.2、缺陷二：可以隐式地转换为整型

枚举的成员总是可以被隐式地转换为整型，如上面的例子，General总是可以被隐式的转换为整型值0

16.1.3、缺陷三：占用的空间大小是一个”不确定量”

C++枚举所基于的”基础类型”是由编译器来决定的，这会导致枚举类型成员的基本类型的不确定性问题，例如：

#include ＜iostream＞
using namespace std;
enum D{D1=1,D2=2,Dbig=0xFFFFFFF0U};
enum E{E1=1,E2=2,Ebig=0xFFFFFFFFFLL};
int main(){
cout＜＜Dbig＜＜endl;//编译器输出不同,g++： 4294967280
cout＜＜sizeof(D1)＜＜endl;//4
cout＜＜sizeof(Dbig)＜＜endl;//4
cout＜＜Ebig＜＜endl;//68719476735
cout＜＜sizeof(E1)＜＜endl;//8
return 0;
}

16.2、强枚举类型

声明强类型枚举非常简单，只需要在enum后加上关键字class。比如：

enum class Type{General,Light,Medium,Heavy};
// 例如
enum class Months:int{Jan, Feb, Mar, Apr, May};
  • 强作用域，强类型枚举成员的名称不会被输出到其父作用域空间，即不能直接被全局可见；
  • 转换限制， 强类型枚举成员的值不可以与整型隐式地相互转换；
  • 可以指定底层基础类型，强类型枚举默认的底层类型为int,但也可以显式地指定底层类型， 具体方法为在枚举名称后面加上”：type”，其中type可以是除wchar_t以外的任何整型；
    如：
enum class Type:char{General,Light,Medium,Heavy};
#include ＜iostream＞
using namespace std;
enum class Type{General,Light,Medium,Heavy};
enum class Category{General=1,Pistol,MachineGun,Cannon};
int main(){
Type t=Type::Light;
t=General;//编译失败， 必须使用强类型名称
if(t==Category::General)//编译失败， 必须使用Type中的General
cout＜＜"General Weapon"＜＜endl;
if(t＞Type::General)//通过编译
cout＜＜"Not General Weapon"＜＜endl;
if(t＞0)//编译失败， 无法转换为int类型
cout＜＜"Not General Weapon"＜＜endl;
if((int)t＞0)//通过编译
cout＜＜"Not General Weapon"＜＜endl;
cout＜＜is_pod＜Type＞::value＜＜endl;//1
cout＜＜is_pod＜Category＞::value＜＜endl;//1
return 0;
}
